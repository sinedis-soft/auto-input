# Windows App UI/UX Design

Помогай разрабатывать Windows-приложения так, чтобы они выглядели аккуратно, современно, нативно для Windows 11 и были удобны реальному пользователю. Не ограничивайся рабочим кодом: оценивай понятность сценария, состояния, ошибки, отступы, доступность и диагностируемость.

## Главный принцип

Делай спокойный, чистый, предсказуемый и практичный интерфейс. Дизайн должен помогать пользователю быстро понять:

- что сейчас происходит;
- что нужно заполнить;
- какая операция запущена;
- где ошибка;
- какое действие главное;
- что можно сделать дальше.

Не добавляй украшения ради украшений.

## Визуальный стиль

Ориентируйся на Windows 11, Fluent Design и WinUI-подход.

Используй:

- мягкие скругления;
- достаточные отступы;
- понятную иерархию заголовков;
- нейтральные цвета;
- аккуратные карточки или группы;
- понятные кнопки;
- единообразные размеры элементов;
- стабильную сетку;
- нормальный dark/light mode, если это поддерживается выбранным стеком.

Не используй кислотные цвета, случайные градиенты, мелкий текст, перегруженные панели, много разных шрифтов, огромные иконки без смысла, бесполезные анимации и самодельные контролы там, где есть стандартные.

## Технологический подход

- Если проект на C#/.NET, предпочитай WinUI 3 / Windows App SDK для современного нативного Windows-интерфейса.
- Если проект на Python, предпочитай PySide6 / PyQt6 для серьезного GUI; CustomTkinter допустим для простой внутренней утилиты; Tkinter без стилизации используй только для прототипа.
- Если текущий проект уже использует конкретный GUI-фреймворк, не меняй стек без необходимости. Сначала улучши структуру, отступы, состояния, тексты, ошибки и навигацию.

## Структура окна

Каждый экран должен иметь понятную композицию:

- верхняя зона — название экрана и краткое описание;
- центральная зона — основное действие или форма;
- нижняя зона — статус, прогресс, вторичные действия;
- одна главная кнопка — визуально понятная;
- опасные действия — отделены и требуют подтверждения.

Если функций мало, используй одно окно с четкими блоками. Если функций много, используй боковую навигацию или вкладки: «Главная», «Заявки / Обработка», «Автоматизация», «Настройки», «Логи», «О программе».

## Отступы, размеры и типографика

Соблюдай визуальный ритм:

- 8 px — между близкими элементами;
- 12 px — между подписью и группой;
- 16 px — внутренний отступ карточки или панели;
- 24 px — между крупными блоками;
- 32 px — внешний отступ страницы, если окно достаточно большое.

Используй один системный шрифт или близкий к Windows UI: Segoe UI / Segoe UI Variable. Делай короткие заголовки, полезные подзаголовки, ясные labels и ошибки рядом с проблемным полем. Не используй ALL CAPS для обычных надписей.

## Цвета и доступность

Цвет должен объяснять, а не украшать:

- нейтральный фон;
- один основной акцентный цвет;
- зеленый — успех;
- желтый/оранжевый — предупреждение;
- красный — ошибка или опасное действие;
- серый — вторичная информация.

Не передавай смысл только цветом: добавляй текст или иконку. Проверяй контраст обычного текста не ниже 4.5:1, читаемый размер шрифта, focus state, tab order, подписи у полей и состояния disabled/loading/error/success. Placeholder не должен быть единственной подписью поля.

## Формы

Группируй поля по смыслу: «Данные клиента», «Данные автомобиля», «Параметры полиса», «Документы», «Служебные данные». Для каждого поля указывай понятный label. Валидируй до отправки, показывай ошибки рядом с полем и не очищай форму после ошибки.

## Кнопки и действия

На экране должна быть одна главная кнопка с конкретным текстом: «Запустить обработку», «Сохранить настройки», «Отправить в Bitrix24», «Сформировать отчет». Вторичные действия называй конкретно: «Отмена», «Проверить подключение», «Открыть лог», «Очистить». Опасные действия вроде «Удалить», «Сбросить настройки», «Очистить все данные» визуально отделяй и подтверждай.

## Состояния приложения

Для долгих операций обязательно реализуй состояния:

- `idle` — ничего не запущено;
- `loading` / `running` — операция выполняется;
- `success` — операция завершена;
- `warning` — есть проблема, но работа возможна;
- `error` — операция не выполнена;
- `cancelled` — пользователь остановил процесс.

Если операция длится больше 1–2 секунд, показывай индикатор выполнения или статус. Для автоматизации, роботов и интеграций показывай текущий шаг, сколько объектов обработано, где произошла ошибка и можно ли повторить операцию. Долгие операции запускай в фоне; не блокируй главное окно без причины; блокируй только опасные повторные действия; добавляй «Остановить», если процесс может идти долго.

## Логи и диагностика

Для внутренних Windows-утилит лог — часть интерфейса. Сделай отдельную область «Журнал» или «Диагностика». В логах показывай время, уровень (`info` / `warning` / `error`), краткое сообщение и детали по кнопке «Подробнее». Не показывай traceback по умолчанию; оставляй его в расширенной диагностике или файле лога.

## Настройки и секреты

Разделяй настройки по группам: «Подключение», «Пути к файлам», «Bitrix24», «ASKO/NDOC», «Поведение робота», «Логи», «Безопасность». Токены и пароли скрывай, не выводи секреты в интерфейс и логи, добавляй «Проверить подключение» и понятные сообщения об ошибках.

## Empty states и ошибки

Если данных нет, не показывай пустой экран. Объясни, что сделать дальше, например: «Заявок пока нет. Загрузите файл или получите данные из Bitrix24.» Добавь кнопку основного действия.

Ошибка должна отвечать на три вопроса: что произошло, почему это важно, что делать дальше. Пиши пользовательские сообщения вместо сырых исключений. Технические детали помещай в раскрываемый блок или лог.

## Чек-лист перед завершением UI-задачи

Проверь, что:

- интерфейс выглядит как Windows-приложение, а не черновик;
- есть нормальные отступы и понятная иерархия;
- главное действие видно сразу;
- ошибки понятны пользователю;
- долгие операции имеют статус;
- форма не теряет данные после ошибки;
- есть disabled/loading/success/error состояния;
- нет случайных цветов и шрифтов;
- текст не слишком мелкий;
- можно пользоваться клавиатурой;
- секреты не выводятся в интерфейс и логи;
- темная тема не ломает читаемость, если она есть;
- окно нормально масштабируется.

## Как отвечать при работе над UI

Когда меняешь интерфейс, кратко объясняй:

- что улучшено;
- почему так лучше;
- какие файлы изменены;
- какие состояния добавлены;
- что осталось проверить вручную.

Если видишь плохое UX-решение, прямо предложи более правильный вариант. Если задача требует выбора между быстрым прототипом и нормальным интерфейсом, явно раздели эти варианты.

## Приоритеты

1. Понятность.
2. Стабильность.
3. Нативность.
4. Визуальная полировка.

Красивый интерфейс без понятных ошибок и состояний считай незавершенным.
