# Unified Apps

Папка содержит первые две объединяющие Python-утилиты для разрозненных проектов автоматизации.

## 1. `chrome_capture_app.py`

Универсальное приложение для снимков активной вкладки Chrome. Оператор вводит имя снимка, приложение сохраняет:

- `.png` — скриншот страницы;
- `.txt` — человекочитаемый список видимых полей, кнопок и ссылок;
- `.json` — структурированные данные DOM для дальнейшей настройки селекторов.

Chrome нужно запустить с remote debugging:

```bat
chrome.exe --remote-debugging-port=9222 --user-data-dir="C:\chrome-capture-profile"
python unified_apps\chrome_capture_app.py
```

## 2. `bitrix_policy_router_app.py`


Единое стартовое окно для сделок Bitrix24 и страховых роботов. Старые GUI больше не открываются: код WARTA и ASKO подключается через интегрированные адаптеры внутри нового приложения. В приложении есть обязательные элементы:


- кнопки `Настройки`, `Начать`, `Далее`, `Новый полис`, `Сбросить сценарий`;
- поле `ID DEAL BITRIX или ссылка`, куда можно вставлять через Ctrl+V / Paste;
- область `Ход работы`;
- область `Вводимые данные из Bitrix24`, где текст можно выделить и скопировать;
- область `Журнал`, где текст можно выделить и скопировать.


Сейчас маршрутизатор выполняет интегрированные сценарии без запуска старых окон:

- WARTA — использует worker-логику из `warta_robot_app_v2/warta_worker.py`;
- ASKO — использует Selenium/Bitrix-логику, перенесенную из `asko_bitrix_filler/asko_bitrix_filler.py` в адаптер `unified_apps/scenario_adapters_parts/asko.py`; `unified_apps/scenario_adapters.py` оставлен фасадом совместимости.


Маршрутизация выполняется по полям сделки Bitrix24:

- `UF_CRM_1686683031442` — страховая компания;
- `UF_CRM_1690539097` — продукт страхования;
- `UF_CRM_1700656576088` — страна полиса.

ASKO выбирается при компании `1091 / АО «Страховая Компания «АСКО»`, продукте `425 / ОСГО ВТС нерезидента` и стране `1101 / КАЗАХСТАН`. WARTA выбирается при компании `231 / TUiR WARTA S.A.` и продукте `425 / ОСГО ВТС нерезидента`.

Новые страховые сценарии добавляются через список `SCENARIOS` в `bitrix_policy_router_app.py`, отдельный адаптер в `scenario_adapters_parts/` и фабрику `build_adapter()` в `scenario_adapters_parts/factory.py`: нужно указать обязательные поля `required_fields`, ключевые слова fallback-маршрутизации и новый адаптер выполнения.

## Как менять логины и пароли страховых

1. Запустите единое приложение:

```bat
python unified_apps\bitrix_policy_router_app.py
```

2. Нажмите кнопку `Настройки`.
3. На вкладке `WARTA` заполните или измените:
   - `WARTA URL`;
   - `WARTA login`;
   - `WARTA password`.
4. На вкладке `ASKO` заполните или измените:
   - `ASKO login`;
   - `ASKO password`;
   - при необходимости `Chrome profile dir` и fallback-срок, который используется только если срок в Bitrix пустой.
5. Нажмите `Сохранить настройки`. Новые данные будут использованы при следующем запуске сценария. Для ASKO срок страхования берётся из поля Bitrix `UF_CRM_1686152209741`, а настройка срока в приложении остаётся запасным вариантом.


Настройки хранятся локально в `unified_apps/bitrix_policy_router.settings.json`. Не отправляйте этот файл в чат и не передавайте его посторонним, потому что внутри могут быть webhook, логины и пароли. Если пароль изменился в кабинете страховой, просто откройте `Настройки`, замените пароль на соответствующей вкладке и сохраните.

Запуск:

```bat
python unified_apps\bitrix_policy_router_app.py
```

## 3. Новый PySide6 UI

Новый профессиональный Windows-like интерфейс находится в `unified_apps/pyside_app.py`. Он не удаляет старый Tkinter router и использует тот же файл настроек:

- настройки: `unified_apps/bitrix_policy_router.settings.json`;
- новый UI: `python -m unified_apps.pyside_app` или `python unified_apps\pyside_app.py`;
- старый fallback UI: `python unified_apps\bitrix_policy_router_app.py`.

Перед запуском установите зависимости:

```bat
pip install -r requirements.txt
```

В новом UI доступны разделы «Сделка», «ASKO», «WARTA», «Журнал» и «Настройки». Долгие операции Bitrix/Selenium/WARTA запускаются в фоне через Qt worker, а события сценариев попадают в верхний статус и общий журнал.

Для ASKO сохранена существующая Selenium/Bitrix-логика адаптера. После заполнения основных полей приложение больше не нажимает «Далее» автоматически: оператор проверяет данные и вручную нажимает «Далее» в ASKO, затем нажимает «Далее» в приложении для добавления застрахованного юрлица.
